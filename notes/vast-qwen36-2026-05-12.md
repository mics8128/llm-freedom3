## Earlier failed/learned constraints

### Driver requirements

`vllm 0.20.2+cu129` / `torch 2.11.0+cu129` fails on driver 575.x even though Vast reports CUDA 12.9:

```text
RuntimeError: The NVIDIA driver on your system is too old (found version 12090)
```

Need driver >=580 in practice.

### Compute capability

Quadro RTX 8000 / compute capability 7.5 is bad:

- no BF16 support, fallback float16
- FA2 unsupported (`>=8` required)
- no native FP8
- Qwen3.6 GDN/Mamba init got stuck

Avoid sm75/Turing. Prefer compute capability >=8.9 (4090/L40S/Ada/Hopper).

### Runtime compiler deps

Clean image originally failed on driver 580 host due missing compiler/Python.h:

```text
Failed to find C compiler
fatal error: Python.h: No such file or directory
```

Fixed in Dockerfile by adding compiler deps + `python3.12-dev`.

## Qwen/vLLM recipe findings

Official vLLM Qwen3.6-27B recipe says:

```bash
vllm serve Qwen/Qwen3.6-27B-FP8 \
  --max-model-len 262144 \
  --reasoning-parser qwen3
```

MTP official example:

```bash
vllm serve Qwen/Qwen3.6-27B-FP8 \
  --speculative-config '{"method": "mtp", "num_speculative_tokens": 1}' \
  --reasoning-parser qwen3
```

Text-only example:

```bash
vllm serve Qwen/Qwen3.6-27B-FP8 \
  --language-model-only \
  --reasoning-parser qwen3 \
  --enable-prefix-caching
```

Troubleshooting note:

- Prefix caching in Mamba `align` mode is experimental.
- For long contexts, MTP >1 may be unstable; start with `num_speculative_tokens=1`.
- For CUDA graph / Mamba cache size error, reduce `--max-cudagraph-capture-size`.

## Caveat on target

Target `>=100 tok/s` with `192K context`, `image enabled`, `2–4 concurrency` on single RTX 4090 is probably unrealistic because KV cache limits force concurrency near 1 at 192K. Two separate 4090 instances could load-balance independent requests, but each single engine likely cannot do 2–4 true concurrent 192K requests.
